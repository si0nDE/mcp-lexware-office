# MCP Lexware Office — v1.0 Design Spec

**Datum:** 2026-04-19  
**Ziel:** Vollständige Lexware-Office-API-Abdeckung + Testinfrastruktur → v1.0.0

---

## Kontext

Der MCP-Server steht bei v0.5.0 mit 43 Tools. Alle CRUD-Operationen für Rechnungen, Angebote, Gutschriften, Auftragsbestätigungen, Lieferscheine, Mahnungen (nur Write), Belege, Artikel und Kontakte sind implementiert. Ziel ist es, bis v1.0 alles abzudecken, was die Lexware-Office-API hergibt, und dabei einen "Buchhaltungsmitarbeiter-Ersatz" über Claude zu ermöglichen.

---

## Section 1: Neue Tools (API-Vollständigkeit)

10 neue Tools bringen die Abdeckung auf 100%.

| Tool | Methode | Endpunkt | Prio |
|---|---|---|---|
| `get-dunnings` | GET | `/v1/dunnings` | 🔴 |
| `get-dunning-details` | GET | `/v1/dunnings/{id}` | 🔴 |
| `get-contact-details` | GET | `/v1/contacts/{id}` | 🔴 |
| `list-event-subscriptions` | GET | `/v1/event-subscriptions` | 🟡 |
| `create-event-subscription` | POST | `/v1/event-subscriptions` | 🟡 |
| `get-event-subscription` | GET | `/v1/event-subscriptions/{id}` | 🟡 |
| `delete-event-subscription` | DELETE | `/v1/event-subscriptions/{id}` | 🟡 |
| `get-document-file` | GET | `/v1/{doctype}/{id}/file` | 🟡 |
| `upload-file` | POST | `/v1/files` | 🟢 |
| `upload-file-to-voucher` | POST | `/v1/vouchers/{id}/files` | 🟢 |

### Implementierungsdetails

**`get-document-file`:** Ein einziges Tool mit `docType`-Parameter (enum: `invoices`, `credit-notes`, `quotations`, `order-confirmations`, `delivery-notes`, `dunnings`, `down-payment-invoices`). Nutzt den aktuellen `/file`-Endpunkt — **nicht** den deprecated `/document`-Endpunkt. Gibt PDF oder XML zurück analog zu `get-file`.

**`get-dunnings`:** Listet Mahnungen mit Filteroptionen (z.B. nach Kontakt-ID, Zeitraum). Analog zu `get-invoices`.

**`get-contact-details`:** Gibt einen einzelnen Kontakt per ID zurück. Fehlt bisher komplett — wichtig für Kundenverwaltungs-Workflows.

**`upload-file` / `upload-file-to-voucher`:** Nimmt base64-kodierten Dateiinhalt + Mimetype entgegen. Für Belegerfassung mit Anhang.

**Event Subscriptions:** Vollständiges CRUD für Webhook-Subscriptions. Wichtig für Automatisierungs-Szenarien (z.B. Benachrichtigung bei neuer Zahlung).

### Bewusst nicht implementiert
- `GET /v1/{doctype}/{id}/document` — deprecated, `/file` wird stattdessen verwendet
- `GET /v1/down-payment-invoices` (Liste) — existiert nicht in der Lexware-Office-API

## Section 2: Test-Infrastruktur

### Automatisierte Tests (Vitest + echte API)

Kein Mocking — alle Tests laufen gegen die echte Lexware-Office-API. GET-Operationen dürfen auf echte Daten zugreifen. Schreibende Operationen (POST/PUT/DELETE) ausschließlich mit Testdaten.

**Namenskonvention Testdaten:**
- Dokumente/Artikel: Titel beginnt mit `[TEST]` — z.B. `[TEST] Rechnung Wave 3`
- Kontakte: Name beginnt mit `[TEST]` — z.B. `[TEST] Testkunde`
- Belege: Beschreibung beginnt mit `[TEST]`

**Verzeichnisstruktur:**
```
tests/
  setup.ts                      # API-Key aus env, shared helpers
  teardown.ts                   # Löscht/archiviert alle [TEST]-Ressourcen
  contacts.test.ts
  invoices.test.ts
  quotations.test.ts
  credit-notes.test.ts
  order-confirmations.test.ts
  delivery-notes.test.ts
  dunnings.test.ts
  vouchers.test.ts
  articles.test.ts
  event-subscriptions.test.ts
  files.test.ts
```

**Testaufbau pro Datei:**
1. `beforeAll` — erstellt Testressource mit `[TEST]`-Prefix
2. Tests — lesen, prüfen, ggf. Status-Übergänge (z.B. finalisieren)
3. `afterAll` — löscht oder archiviert die Testressource

**CI:** Kein automatischer API-Key in CI. Tests laufen lokal via `npm run test:integration`. Der API-Key wird aus einer `.env`-Datei gelesen (nicht committet).

### Manuelle Abnahme (Claude Desktop)

Vor dem v1.0-Tag führt der User die wichtigsten Workflows live in Claude Desktop durch. Eine Checkliste in `docs/testing-checklist.md` dokumentiert die Szenarien:

- Angebot erstellen → Auftragsbestätigung → Rechnung → Mahnung → PDF abrufen
- Kontakt anlegen → Details abrufen → Rechnung zuordnen
- Beleg erfassen + Datei hochladen
- Event Subscription anlegen → testen → löschen
- Wiederkehrende Vorlage abrufen

## Section 3: Release & Upstream-Strategie

### Versioning

| Version | Inhalt |
|---|---|
| `v0.5.0` | Aktueller Stand (43 Tools) |
| `v0.6.0` | + 10 neue Tools (API-vollständig) |
| `v1.0.0` | + Tests bestanden, manuelle Abnahme abgeschlossen |

### Branch-Strategie

```
main (v0.5.0)
  └── claude/wave-3-api-completeness   ← Implementierung der 10 Tools
        └── PR → main → v0.6.0 tag
main (v0.6.0)
  └── claude/testing-infrastructure    ← Vitest-Tests + Checkliste
        └── PR → main → v1.0.0 tag
```

### Upstream zu Jannik Wempe

- Erst wenn v1.0.0 lokal vollständig getestet und sauber → PR an `JannikWempe/mcp-lexware-office`
- Format: ein gebündelter PR oder zwei separate (Tools + Tests) — nach Jannik's Präferenz
- CHANGELOG.md und README Tool-Tabelle werden vor dem PR aktualisiert

## Implementierungsreihenfolge

1. `get-contact-details` — einfachstes Tool, sofort nützlich
2. `get-dunnings` + `get-dunning-details` — komplettiert den Mahnungskreislauf
3. `get-document-file` — PDF-Download für alle Dokumenttypen
4. Event Subscriptions (list, create, get, delete)
5. `upload-file` + `upload-file-to-voucher`
6. Tests schreiben (parallel zu Implementierung wo möglich)
7. Manuelle Abnahme → v1.0.0 tag
